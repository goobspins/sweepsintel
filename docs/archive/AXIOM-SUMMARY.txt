DATABASE SCHEMA FINALIZED
─────────────────────────────────────────────────────────────────────────────

7 tables designed (ready to deploy to Neon):

1. CASINOS (Master data + affiliate links)
   - Stores 33+ casino profiles, wash games, ratings, affiliate URLs
   - Indexes optimized for sorting by rating/tier

2. DAILY_BONUS_CLAIMS (User bonus tracking)
   - One row per user per casino per day
   - Drives retention engine (daily habit forming)
   - Tracks affiliate clicks (soft conversion)

3. PL_LEDGER (Expense/earning tracking)
   - Users log offer spend, SC earned, redemptions, fees
   - Per-casino profitability analysis
   - Tax-preparation CSV export

4. BAN_REPORTS (Community UGC)
   - User-submitted ban/issue reports
   - IP hash + email hash for deduplication
   - is_flagged field for moderation (not auto-published if duplicate)
   - Indexes for fast published-reports query

5. BAN_UPTICK_ALERTS (Signal aggregation)
   - Hourly job aggregates ban reports by casino + issue_type
   - Confidence scoring (unique IPs / threshold ^ exponent)
   - Broadcast trigger when confidence > 0.70
   - Automatic alert generation + notification delivery

6. AUTH_SESSIONS (Email OTP state)
   - User email, OTP hash, session token
   - 60-day session expiry
   - Indexes for fast email + token lookups

7. CLICKS (Affiliate link tracking)
   - Logs casino_id, user_id (if logged in), timestamp
   - Referrer source (tracker, casino dir, search, etc.)
   - User agent for bot detection
   - Powers affiliate revenue reporting dashboard

All tables have optimal indexes for query patterns.
All constraints (UNIQUE, PRIMARY KEY, FOREIGN KEY) defined.
Ready to deploy: `psql < docs/NEON_SCHEMA.sql`

CRITICAL DESIGN DECISIONS MADE
─────────────────────────────────────────────────────────────────────────────

1. DAILY BONUS TRACKER AS RETENTION ENGINE
   Mechanism: Users return daily to check/claim bonuses across 33 casinos.
   When they click to a new casino, affiliate link fires (soft conversion).
   This is the primary affiliate funnel, not a secondary feature.

   Design: Prominent UI, badge showing "claimed/unclaimed" status per casino.
   Primary CTA: "Sign Up & Claim Daily Bonus" for casinos not yet joined.

2. BAN REPORT DEDUPLICATION (IP-BASED FLAGGING)
   Problem: Without dedup, spam/duplicates flood the feed, degrading signal.
   Solution: On submission, hash reporter IP + check last 7 days:
     - If 3+ reports from same IP → flag (not auto-published)
     - Dylan (admin) reviews flagged submissions in moderation queue
     - Only unflagged reports auto-publish to public feed

   This prevents spam while maintaining low friction for real reports.

3. AFFILIATE LINKS EMBEDDED AT BUILD TIME
   Why: Avoid database query per page load, reduce latency.
   How: Affiliate URLs stored in Neon casinos table; at Astro build time,
        frontmatter is populated from DB, embedded in static HTML.

   Update flow: Dylan provides new link → Collective updates DB → redeploy
   (full redeploy takes 1-2 min via Vercel)

4. ASTRO CONTENT COLLECTIONS FOR CASINO PROFILES
   Why: Type-safe frontmatter (rating must be 0-10), auto-indexing, fast builds.
   How: Each casino is a .mdx file; Astro validates schema, allows queries.

   Result: `/casinos/[slug].astro` page renders dynamically at request time,
   pulling content from markdown + metadata from DB (affiliate links, stats).
